# Agent guide — gnade_app

## Project

**gnade_app** — A business management app for African small business owners covering:
Sales & Invoicing · Inventory · Expenses · Reports · Customer & Debt Management · Team Management · Receipt Printing (Bluetooth & WiFi thermal) · Push Notifications (FCM + Supabase)

Generated by [FlutterInit](https://flutterinit.com). Package: `com.example.gnade_app`.

## Architecture (`feature-first`)

```text
lib/src/
├── features/
│   ├── auth/              # login, signup, forgot password
│   ├── home/              # dashboard — sales summary, quick actions, low stock
│   ├── onboarding/        # splash, onboarding flow
│   ├── sales/             # POS, select item, new sale, sale detail, sales history
│   ├── inventory/         # product list, add/edit product, stock management
│   ├── expenses/          # expense list, log expense form
│   ├── reports/           # sales report, profit report, analytics charts
│   ├── customers/         # customer list (all/debtors/depositers), customer detail
│   ├── team/              # team list, add member, role management
│   ├── printing/          # receipt preview, print via BT/WiFi/PDF, printer settings
│   └── notifications/     # notification centre, notification settings
├── shared/                # cross-feature widgets & utils only
├── routing/
├── services/
├── config/
├── theme/
├── imports/
├── extensions/
└── utils/
```

**Rules**

- Each feature is self-contained — no imports from another feature's internals.
- Shared code lives only under `lib/src/shared/`.
- Feature folder structure per feature:
  ```
  <feature>/
  ├── data/
  │   ├── models/          # *_model.dart — Supabase JSON mapping
  │   └── repositories/    # *_repository_impl.dart — concrete Supabase calls
  ├── domain/
  │   ├── entities/        # pure Dart entities, no Flutter deps
  │   └── repositories/    # abstract repository interfaces
  └── presentation/
      ├── providers/        # Riverpod providers for this feature
      ├── screens/          # *_screen.dart files
      └── widgets/          # feature-scoped widgets (not shared)
  ```

## State Management (Riverpod)

- Use `ConsumerWidget` / `ConsumerStatefulWidget` where reactive state is needed.
- Providers live in `presentation/providers/` inside the feature folder.
- `ref.watch` inside `build()`; `ref.read` inside callbacks and one-off actions — **never** `ref.read` in `build()`.
- Do not nest a second `ProviderScope` — one at app root is already configured.
- This project does **not** use `@riverpod` / `riverpod_generator` — define providers manually.

### Provider conventions

```dart
// Repository provider
final saleRepositoryProvider = Provider<SaleRepository>((ref) {
  return SaleRepositoryImpl(Supabase.instance.client);
});

// Async data provider
final salesProvider = FutureProvider.family<List<Sale>, String>(
  (ref, businessId) => ref.read(saleRepositoryProvider).getSales(businessId),
);

// Mutable state
final selectedPrinterProvider = StateNotifierProvider<SelectedPrinterNotifier, PrinterDevice?>(
  (_) => SelectedPrinterNotifier(),
);
```

## Navigation

- All routes are defined in `lib/src/routing/app_router.dart` — do not scatter route tables elsewhere.
- Route name constants live in `lib/src/routing/app_routes.dart`.
- Prefer typed/named routes — avoid hard-coded path strings in widgets.
- `context.go()` replaces the stack; `context.push()` pushes on top.
- Redirects and guards belong in the router, not inside individual screens.
- Pass complex objects (e.g. `ReceiptData`) via `extra:`, not query params.

## Backend (Supabase)

- All Supabase calls go through repositories — never from UI or providers directly.
- Assume Row Level Security on all tables — never bypass RLS in client code.
- Session persistence is handled by the SDK — avoid manual token caching.
- Configure URL and anon key via `.env` — see **SETUP.md**.

### Database tables

| Table               | Purpose                                                         |
| ------------------- | --------------------------------------------------------------- |
| `businesses`        | Business profile — name, logo, currency, address                |
| `users`             | App users — role (`owner` / `manager` / `staff`), business_id   |
| `products`          | Inventory items — cost/sell price, quantity, low_stock_at, unit |
| `sales`             | Sale records — total, paid, status (`paid`/`partial`/`debt`)    |
| `sale_items`        | Line items per sale (product snapshot at time of sale)          |
| `expenses`          | Expenses — title, category, amount, date                        |
| `device_tokens`     | FCM tokens per user/device for push notifications               |
| `notification_logs` | In-app notification history                                     |

### Supabase RPC functions

```sql
-- Atomically decrement product stock on sale
decrement_stock(product_id uuid, qty int)
```

### Edge Functions (supabase/functions/)

| Function            | Trigger                    | Purpose                               |
| ------------------- | -------------------------- | ------------------------------------- |
| `send-notification` | Called by DB triggers      | FCM HTTP v1 push via Google Auth      |
| `daily-summary`     | pg_cron at 20:00 daily     | End-of-day sales summary notification |
| `debt-reminder`     | pg_cron every Monday 09:00 | Outstanding customer balance alert    |

### DB triggers (notify on events)

| Trigger          | Table      | Event           | Notification         |
| ---------------- | ---------- | --------------- | -------------------- |
| `on_new_sale`    | `sales`    | INSERT          | 🛒 New Sale Recorded |
| `on_low_stock`   | `products` | UPDATE quantity | ⚠️ Low Stock Alert   |
| `on_new_expense` | `expenses` | INSERT          | 💸 Expense Logged    |

## Networking

- Use the shared `Dio` instance from `lib/src/config/app_config.dart` — never `Dio()` inside a widget or screen.
- Extend or use existing services under `lib/src/services/` for HTTP calls.
- Map transport errors to `Failure` / `FutureEither` results via `runTask()` — do not leak raw `DioException` to UI.

---

## Services & Shared Conventions

- Services are singletons with `ClassName.instance`, returning `FutureEither<T>` via `runTask()`.
- Never pass `BuildContext` into services — use `rootContext` from the global navigator helper when UI is required.
- Logging: `AppLogger`; user feedback: `showGlobalToast()`; dialogs: `showAppDialog()` / `context.showAppDialog()`.
- Primary import barrel: `package:gnade_app/src/imports/imports.dart`.
- File names: `snake_case.dart`; classes: `PascalCase`; private members: `_camelCase`.
- Prefer `const` constructors where possible; avoid `dynamic` without justification.
- No empty `catch` blocks — log, map to failure, or rethrow.
- Do not add packages that duplicate existing stack choices.

### Dart / Flutter anti-patterns

- No business logic in widget `build()` methods.
- No direct backend or network calls from presentation widgets.
- Do not use `ref.read` inside `build()`.
- Do not write entire screens as monolithic files. A screen's `build()` method must only compose named child widgets — never contain inline widget trees longer than ~30 lines. Extract into `presentation/widgets/<feature>_<widget_name>.dart` when: (a) a widget has its own state, (b) a section is visually distinct (card, tile, sheet, form, chart), or (c) a widget exceeds 60 lines. The screen file itself should not exceed 100–120 lines.

## Features Reference

### Sales (`features/sales/`)

- `Sale` entity: id, business_id, customer_name, total_amount, amount_paid, discount, payment_method, status, invoice_no, items
- `SaleItem` entity: product snapshot (name, qty, unit_price, total)
- Payment methods: `cash` | `transfer` | `credit`
- Sale status: `paid` | `partial` | `debt`
- After creating a sale: call `decrement_stock` RPC for each item, then navigate to `/print-receipt`
- Invoice numbers are auto-generated (format: `BZ` + 8-digit random)

### Inventory (`features/inventory/`)

- `Product` entity: id, name, sku, category, cost_price, sell_price, quantity, low_stock_at, unit, expiry_date
- Units: `pcs` | `carton` | `bottle` | `crate` | `litre` | `kg`
- Stock states: high stock (green) · low stock (amber, qty ≤ low_stock_at) · finished (red, qty = 0)
- `isLowStock` computed: `quantity <= lowStockAt`
- Dashboard home screen shows low stock items — pull from products where quantity ≤ low_stock_at

### Expenses (`features/expenses/`)

- Categories: `rent` | `salary` | `utilities` | `supplies` | `transport` | `other`
- Linked to business_id and created_by (user_id)

### Customers (`features/customers/`)

- Customer types: All · Debtors (have unpaid sales) · Depositers (advance payment)
- `I am owed` summary = sum of (total_amount - amount_paid) across all partial/debt sales for this customer
- Customer detail shows: purchase history, debt records, notes tab

### Team (`features/team/`)

- Roles: `owner` | `manager` | `staff`
- Role-based access (check before rendering sensitive actions):

```dart
enum UserRole { owner, manager, staff }

extension RolePermissions on UserRole {
  bool get canDeleteSales   => this == UserRole.owner;
  bool get canViewReports   => this != UserRole.staff;
  bool get canManageTeam    => this == UserRole.owner;
  bool get canAddProducts   => this != UserRole.staff;
  bool get canViewFinance   => this != UserRole.staff;
}
```

### Receipt Printing (`features/printing/`)

- `ReceiptData` entity: pure Dart, no Flutter deps — business info, invoice no, items, totals, payment method
- `PrinterDevice`: id (MAC or IP), name, type (`bluetooth` | `network` | `pdf`), paperWidth (58 or 80 mm)
- `PrinterRepository`: abstract — `discoverPrinters()`, `connect()`, `printReceipt()`, `disconnect()`
- Implementations: `BluetoothPrinterRepository` (esc_pos_bluetooth) · `NetworkPrinterRepository` (esc_pos_network, TCP port 9100) · `PdfPrinterRepository` (pdf + printing)
- Printer preference is persisted via `SharedPreferences` (`printer_id`, `printer_name`, `printer_type`)
- Active printer type controlled by `printerTypeProvider` (StateProvider)
- After completing a sale, navigate to `/print-receipt` with `extra: receiptData`
- If a saved printer exists, offer one-tap auto-print without navigating away

#### ESC/POS receipt structure

```
[Business name — bold, centered, size2]
[Address · Phone — centered]
[HR divider]
[Invoice # · Date · Customer]
[HR divider]
[Item | Qty | Price — columns]
[--- dashes ---]
[per item rows]
[HR divider]
[Discount (if any)]
[TOTAL — bold, size2]
[Paid · Balance (if partial)]
[HR divider]
[Payment method · PAID IN FULL stamp]
[Thank you footer]
[feed 2 · cut]
```

### Push Notifications (`features/notifications/`)

- `NotificationService` is a singleton initialized in `app.dart` via `addPostFrameCallback`
- FCM token is saved to `device_tokens` table on login and on token refresh
- Foreground messages shown via `flutter_local_notifications` with per-event channels
- Tapping a notification navigates to the relevant route via `AppRouter.navigatorKey`

#### Notification channels (Android)

| Channel ID  | Name           | Events                       |
| ----------- | -------------- | ---------------------------- |
| `sales`     | Sales          | New sale recorded            |
| `low_stock` | Low Stock      | Product quantity ≤ threshold |
| `expenses`  | Expenses       | Expense logged               |
| `summary`   | Daily Summary  | End-of-day report            |
| `debt`      | Debt Reminders | Weekly outstanding balance   |

#### Notification → route mapping

| Event          | Route on tap         |
| -------------- | -------------------- |
| New sale       | `/sales/:id`         |
| Low stock      | `/inventory/:id`     |
| Expense logged | `/expenses`          |
| Daily summary  | `/reports`           |
| Debt reminder  | `/sales?filter=debt` |

## How to Add a New Feature

1. Create `lib/src/features/<feature>/` with `data/`, `domain/`, `presentation/` subfolders.
2. Define the domain entity in `domain/entities/` — pure Dart only.
3. Define the abstract repository in `domain/repositories/`.
4. Implement the repository in `data/repositories/` using Supabase.
5. Create a data model in `data/models/` with `fromMap()` / `toMap()`.
6. Add Riverpod providers in `presentation/providers/`.
7. Build screens in `presentation/screens/`.
8. Register routes in `lib/src/routing/app_router.dart`.
9. Add route name constants to `lib/src/routing/app_routes.dart`.
10. Export new files from `lib/src/imports/imports.dart` as needed.

| --------------------------------- | ------------------------------------------------------------------------ |
| `lib/src/routing/app_router.dart` | Central navigation — breaks deep links if wrong                          |
| `lib/main.dart`                   | Initialization order (Firebase, dotenv, Supabase, localization, flavors) |

## Initialization Order (`main.dart`)

```dart
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await dotenv.load();                                    // 1. env vars
  await EasyLocalization.ensureInitialized();             // 2. i18n
  await Firebase.initializeApp(                           // 3. Firebase (FCM)
    options: DefaultFirebaseOptions.currentPlatform,
  );
  await Supabase.initialize(                              // 4. Supabase
    url: dotenv.env['SUPABASE_URL']!,
    anonKey: dotenv.env['SUPABASE_ANON_KEY']!,
  );
  FirebaseMessaging.onBackgroundMessage(                  // 5. FCM background handler
    _firebaseBackgroundHandler,
  );
  runApp(const ProviderScope(child: App()));
}

@pragma('vm:entry-point')
Future<void> _firebaseBackgroundHandler(RemoteMessage message) async {
  await Firebase.initializeApp();
}
```

## Hard Limits

- Do not disable `flutter_lints` rules without an explanatory comment.
- Do not call backend or networking code directly from widgets.
- Do not use `ref.read` inside `build()`.
- Do not commit `.env` files containing production secrets.
- Do not introduce a second state-management or routing library.
- Do not add packages that duplicate existing stack choices.
- Do not bypass Supabase RLS from client code.
- Do not call `decrement_stock` RPC more than once per sale — it is not idempotent.
- For platform setup and API keys, use **[SETUP.md](SETUP.md)**.
- For UI tokens and spacing, use **[DESIGN.md](DESIGN.md)**.
